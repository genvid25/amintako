# Облачный бэкенд конвейера кадров — план

Спутник файла `docs/supabase-schema.sql`. Здесь: где лежат картинки и почему не
в Supabase, как устроен доступ сотрудников, как режиссёру завести проект и что
предстоит сделать в дашборде и воркере.

**Короткая версия архитектуры:**

```
Supabase       -> только данные: очередь кадров, статусы, правки, пути к
                  картинкам, лента событий воркера, доступ      (килобайты)
Репозиторий    -> сами картинки в WebP, раздаются через GitHub Pages (сотни МБ)
Машина воркера -> исходные PNG, в репозиторий не попадают        (гигабайты)
```

---

## 1. Что уже проверено

Схема из `supabase-schema.sql` не «написана на глаз» — она развёрнута на
настоящем PostgreSQL 17 и прогнана 16 сценариями. Проверено:

| Что проверяли | Результат |
|---|---|
| Скрипт выполняется целиком, без ошибок | да |
| Повторный запуск ничего не ломает и **не меняет уже выданные ссылки** | да |
| Скрипт работает, даже если схемы `storage` в проекте нет вовсе | да — бакет не нужен |
| Загрузка настоящей раскадровки (166 кадров, 11 сцен) | да |
| Правило мастер-кадра: 11 первых кадров в очередь, 155 в ожидание | да |
| Одобрение мастер-кадра открывает свою сцену и **не трогает соседние** | да |
| В `scenes.anchor_image` попадает путь в репозитории, а не ссылка на Storage | да |
| Мастер-кадр на переделку — сцена запирается, референс снимается | да |
| Windows-пути с обратными слешами хранятся без искажений | да |
| Посторонний с публичным ключом читает кадры | да (так и задумано) |
| Посторонний меняет статус / промт / **подменяет путь к картинке** / удаляет | **запрещено на всех путях** |
| Посторонний читает токены сотрудников | **запрещено** |
| Подобранный, пустой или отозванный токен | отклоняется |
| Аниматор лезет в чужую серию | отклоняется; режиссёр — проходит |
| Токен и в заголовке, и параметром функции | оба работают |
| Повторная синхронизация раскадровки не затирает статусы и правки | да — при правильном запросе (§5.8) |

Отдельно отмечу строку про подмену пути: теперь, когда картинки лежат снаружи,
важно, что посторонний не может перенаправить кадр на чужую картинку. Проверено —
запись в `versions` закрыта полностью.

## 2. Где лежат картинки и почему не в Supabase

### 2.1 Замеры на реальных кадрах

Я измерил 28 готовых кадров проекта и пересжал их ровно так, как это будет
делать воркер (Pillow, спека: превью 800 px q80, полный 2K q90):

| Формат | Размер кадра | Назначение |
|---|---|---|
| PNG 2752×1536 — как сейчас | **2.42 МБ** | остаётся на машине воркера |
| WebP q90, тот же размер | **403 КБ** | в репозиторий: по клику и на скачивание |
| WebP q80, ширина 800 px | **50 КБ** | в репозиторий: сетка дашборда |

**WebP сжимает кадр в 6.1 раза.** «Без потери качества» — не на слово: сравнил
попиксельно, среднее отличие **42.7 дБ PSNR**, худший кадр 41.3 дБ при пороге
различимости глазом около 40 дБ. Положить рядом и найти отличие не выйдет.

Серия = 166 кадров. С запасом +40 % на переделки (≈232 картинки):

| | 1 серия | 4 серии | В 1 ГБ влезет |
|---|---|---|---|
| PNG, как сейчас | 562 МБ | 2.20 ГБ | 1.8 серии |
| **WebP (полный + превью)** | **103 МБ** | **0.40 ГБ** | **~10 серий** |

### 2.2 Почему Supabase Storage не подходит

Лимиты бесплатного тарифа Supabase (сверено 20.07.2026):

| | Free | Подвох |
|---|---|---|
| Хранилище файлов | 1 ГБ | **на всю организацию**, а не на проект |
| Исходящий трафик | 5 ГБ uncached + 5 ГБ cached | **единый пул на все сервисы** — трафик самой базы ест те же гигабайты |
| Ресайз картинок на лету | **нет** | Image Transformations только с Pro |
| CDN | basic | Smart CDN только с Pro, кэш работает хуже |
| Размер базы | 500 МБ на активный проект | нам хватает с многократным запасом |

Складывать кадры туда нельзя по двум причинам сразу:

1. **Объём.** 4 серии в WebP — 0.40 ГБ из 1 ГБ на организацию. Формально влезает,
   но запас исчезает, а лимит общий на всё, что вы когда-либо заведёте.
2. **Трафик, и это решающее.** 5 ГБ в месяц, общие с трафиком базы. Сетка из 166
   превью — 8 МБ за проход; несколько человек, несколько заходов в день — и лимит
   выбран. При превышении Supabase не «просто предупреждает»: API начинает
   отдавать **402**, а база уходит в read-only, то есть дашборд встаёт.

Плюс на Free нет ресайза на лету, так что превью всё равно пришлось бы делать
самим и хранить отдельными файлами — единственное преимущество Storage отпадает.

### 2.3 GitHub Pages: лимиты и наш бюджет

Картинки остаются там, где они и сейчас, — в репозитории `genvid25/amintako`,
раздаются через GitHub Pages. Лимиты (docs.github.com, проверено 20.07.2026):

| | Лимит | Тип |
|---|---|---|
| Размер опубликованного сайта | **1 ГБ** | **жёсткий** — сайт перестаёт отдаваться |
| Трафик | 100 ГБ/мес | мягкий — GitHub напишет письмо |
| Сборок сайта | 10 в час | мягкий; не действует при своём Actions-workflow |

**По трафику запас двадцатикратный** против Supabase (100 ГБ против 5 ГБ) — это и
есть главный аргумент за такую схему.

Бюджет по объёму считаю честно, потому что 1 ГБ — жёсткий предел, и в него
входит **весь** опубликованный сайт, а не только кадры:

| Что | Сейчас | После 4 серий в WebP |
|---|---|---|
| `Референсы/` (тоже публикуются!) | 94 МБ | 94 МБ + рост |
| `images/` | 70 МБ (PNG) | ~412 МБ (WebP) |
| Остальное (дашборд, данные, скрипты) | ~1 МБ | ~1 МБ |
| **Итого из 1 ГБ** | ~165 МБ | **~510 МБ** |

**Вывод: 4 серии помещаются, остаётся примерно половина гигабайта — ещё серий
на пять.** Но это потолок, и о нём надо помнить заранее, а не упереться в него
посреди работы.

> Побочное наблюдение: папка `Референсы/` (94 МБ) тоже публикуется в интернет и
> ест тот же гигабайт. Если она не должна быть публичной — это отдельный
> разговор, но сейчас она открыта всем, как и кадры.

### 2.4 Риск, о котором легко забыть: история git необратима

Репозиторий теперь работает хранилищем, а у git есть особенность: **удалённый
файл остаётся в истории навсегда.** Сейчас в проекте:

```
рабочая копия images/ + Референсы/   ~164 МБ
.git                                  162 МБ
```

То есть история уже весит столько же, сколько сами файлы. Каждая переделка кадра
добавляет новый файл (`_v2`, `_v3`), и он останется в истории, даже если потом
кадр заменят. Лимит опубликованного *сайта* (1 ГБ) считается по рабочей копии, а
не по истории, поэтому сайт от этого не сломается — но сам репозиторий будет
пухнуть, клонирование замедлится, а GitHub предупреждает при переходе за 1 ГБ и
жёстко режет push на 5 ГБ.

Что с этим делать (по возрастанию радикальности):

1. **Ничего** — при 4 сериях история дорастёт примерно до 0.6 ГБ, это терпимо.
2. **Не коммитить заведомо отбракованные версии** — если кадр переделан трижды,
   в репозитории нужны не все три, а последняя плюс, может быть, одна для
   сравнения. Решает большую часть роста.
3. **Отдельный репозиторий под картинки** (`amintako-images`), чтобы основной
   оставался лёгким и быстро клонировался.
4. **Раз в несколько серий заводить хранилище заново**, а старые серии
   складывать в архив (Google Drive) — история обнуляется.

Рекомендую сразу заложить пункт 2, остальное — по мере необходимости.

### 2.5 Риск: 10 сборок сайта в час

Каждый push в репозиторий запускает пересборку GitHub Pages, а их мягкий лимит —
**10 в час**. Если воркер будет пушить после каждого кадра, на 166 кадрах он
упрётся в лимит и сборки начнут задерживаться.

**Требование к воркеру: пушить пачками, а не покадрово** — например, один push
в конце прогона генерации (по 10–20 кадров). Заодно меньше мусора в истории.

### 2.6 Проект Supabase засыпает через неделю простоя

> «Projects are paused after low activity over a 7-day period» — бесплатный
> проект встаёт на паузу примерно после **недели без обращений к базе**.

- **Данные не теряются.** Восстановление кнопкой **Resume project**, окно —
  **90 дней**. Даже после 90 дней данные не пропадают совсем: из Project Overview
  скачивается логический бэкап, снятый перед паузой.
- **Пока идёт работа, пауза не грозит:** воркер круглосуточно ходит в базу за
  очередью, нескольких запросов в день достаточно.
- **Риск только в долгом простое** — отпуск студии, перерыв между сериями. Тогда
  режиссёр вернётся к неработающему дашборду и должен знать, что чинится это
  одной кнопкой, а не паникой.

Отдельно закладывать «пинг» не нужно — воркер и так его обеспечивает.

### 2.7 Что требуется от воркера: конвертация в WebP

Это ключ ко всей вместимости, без него схема не работает. Машина воркера — на
Windows, воркер уже на Python, поэтому проще всего **Pillow**:

```
pip install Pillow
```

```python
from PIL import Image

im = Image.open(png_path).convert("RGB")

# 1. Полный кадр в репозиторий: 2K, q90 (~400 КБ)
im.save("images/s01_03_v1.webp", "WEBP", quality=90, method=6)

# 2. Превью для сетки дашборда: ширина 800, q80 (~50 КБ)
w = 800
im.resize((w, round(im.height * w / im.width)), Image.LANCZOS) \
  .save("previews/s01_03_v1.webp", "WEBP", quality=80, method=6)

# 3. Исходный PNG НЕ коммитим — оставляем на машине,
#    путь пишем в versions.original_png
```

`method=6` — самое плотное сжатие, около секунды на кадр. На фоне генерации
изображения это незаметно.

**Альтернатива без Python-зависимости** (если Pillow ставить не хочется):
`cwebp.exe` из пакета libwebp Google — один самодостаточный бинарник:

```
cwebp.exe -q 90 -m 6 кадр.png -o images/s01_03_v1.webp
cwebp.exe -q 80 -m 6 -resize 800 0 кадр.png -o previews/s01_03_v1.webp
```

Рекомендую Pillow: воркер и так на Python, а лишний бинарник на Windows — это
ещё один файл, который нужно где-то держать и обновлять.

**Ещё одно требование:** PNG не должны попадать в репозиторий. Понадобится
строка в `.gitignore` (`images/*.png`) и разовая уборка уже закоммиченных 30
файлов — отдельный шаг миграции, см. §6.

### 2.8 Если всё-таки упрёмся

1. **Превью 480 px вместо 800** (22 КБ вместо 50) — вдвое дешевле по трафику,
   в сетке разница почти не видна.
2. **Отдельный репозиторий или архив старых серий** — см. §2.4.
3. **Cloudflare R2** — 10 ГБ бесплатно и **нулевая плата за исходящий трафик**.
   Логичный шаг, если упрёмся в размер сайта, а не в трафик.
4. **Supabase Pro ($25/мес)** — на объём картинок уже не влияет (их там нет),
   зато снимает 7-дневную паузу. Осторожно: **spend cap включён по умолчанию**,
   и с ним перерасход даёт не доплату, а те же 402 и read-only.

## 3. Доступ: кто и как может менять статусы

### 3.1 В чём сложность

Публичный ключ лежит в `dashboard/config.js`, а репозиторий `genvid25/amintako`
**публичный**. Значит, ключ виден любому. Это нормально и так задумано в
Supabase — но только если база сама умеет отличать «смотреть» от «менять».
Иначе любой прохожий проставит 166 кадрам «одобрено» или сотрёт правки.

При этом сотрудники — не технари, а материалы не секретные. Полноценная
авторизация с паролями тут дороже проблемы, которую решает.

### 3.2 Вариант А — личная ссылка с токеном

Каждому сотруднику выдаётся своя ссылка:

```
https://genvid25.github.io/amintako/dashboard/?key=70f60e920e8a4ca98f76dda9cf97af65
```

Дашборд достаёт `key` из адреса и прикладывает к каждой попытке что-то изменить.

**За:** сотруднику не надо делать ничего — открыл закладку и работает. Ни почты,
ни паролей, ни ожидания письма. Токены личные, поэтому видно, кто что одобрил, и
можно отозвать доступ одному, не трогая остальных.

**Против:** ссылка — это и есть пропуск. Переслал в общий чат или показал на
скриншоте — чужой человек получил те же права.

### 3.3 Вариант Б — вход по e-mail (magic link)

**За:** настоящая личность, ссылку сложнее потерять случайно.

**Против:** каждый вход — сходить в почту, найти письмо, вернуться; письма
попадают в спам. Встроенная почта Supabase жёстко лимитирована, для реальной
работы нужен внешний SMTP — **ещё одна регистрация для человека, который не
программист**. Плюс настройка Site URL и Redirect URLs, где легко ошибиться.

### 3.4 Рекомендую вариант А, но усиленный

- Реальный риск здесь — «посторонний испортит статусы», а не «утечёт секрет»:
  кадры и так публично лежат на GitHub Pages.
- Цена ошибки низкая и обратимая: все правки пишутся в ленту событий с именем,
  статус всегда можно вернуть.
- Вариант Б добавляет ежедневное трение всем ради защиты от угрозы, которой в
  этом проекте практически нет.

Чтобы снять главный минус варианта А, в схеме сделано три вещи:

**1. Сотрудник физически не может писать в таблицы.** Права на запись у
публичного ключа отобраны совсем, RLS-политик на запись нет. Единственная
операция — функция `review_frame()`, умеющая ровно две вещи: одобрить и вернуть
на переделку. Даже зная токен, нельзя переписать промт, подменить путь к
картинке или удалить кадр — такого пути не существует. Проверено.

**2. Токен ограничен по серии.** Аниматор ходит только в свои серии, режиссёр —
везде. Проверено.

**3. Токен отзывается одной строкой:**
`update members set active = false where name = 'Амин';`

Дополнительно: одобрение режиссёра помечается отдельно (`final_ok`), поэтому
«аниматор принял» и «режиссёр утвердил окончательно» — разные состояния.

### 3.5 Приятная подробность про секретный ключ

Секретный ключ (`sb_secret_…`) **невозможно использовать из браузера**: Supabase
определяет браузер по заголовку `User-Agent` и отвечает 401. То есть даже если
его случайно вставят в `config.js`, он там не заработает — платформа страхует от
самой опасной ошибки. Не повод его светить, но приятно.

## 4. Как режиссёру завести проект

> Памятка для человека, который не программист. Ничего устанавливать не нужно,
> всё в браузере. Минут 15, из них 2–3 просто ждём.
>
> Supabase иногда меняет вид кнопок. Если что-то называется чуть иначе —
> ориентируйтесь по смыслу, порядок шагов тот же.

### Шаг 1. Завести аккаунт

Откройте **supabase.com** → **Start your project**.

Регистрация двумя способами: через **GitHub** или **обычной почтой с паролем**.
Аккаунт GitHub не нужен — спокойно выбирайте почту. Пароль потребуют посложнее:
не короче 8 символов, обязательно заглавная буква, строчная, цифра и спецсимвол
(`!?<>@#$%`). Входа через Google нет.

### Шаг 2. Создать организацию

Название — любое, например `Эхо в горах`. Тариф **Free**. Карту не спрашивают.

### Шаг 3. Создать проект

- **Project name** — `amintako`.
- **Database password** — нажмите **Generate** и **обязательно сохраните себе**
  (заметки, менеджер паролей). Поменять его потом можно только вручную.
- **Region** — берите **Central EU (Frankfurt)**. Это ближайший к Турции из всех
  доступных: Стамбула, Ближнего Востока и ОАЭ в списке нет вообще. Из
  европейских есть ещё Ирландия, Лондон, Париж, Цюрих, Стокгольм — если
  Франкфурта почему-то не окажется, берите любой, разница только в скорости.

Нажмите **Create new project** и подождите пару минут.

### Шаг 4. Выполнить скрипт базы

1. Левое меню → **SQL Editor** → **New query**.
2. Откройте `docs/supabase-schema.sql`, выделите **весь** текст (Cmd+A / Ctrl+A),
   скопируйте и вставьте в поле.
3. Нажмите зелёную кнопку **Run** справа вверху (или Ctrl+Enter / Cmd+Enter).

Внизу появится табличка:

| Кто | Роль | Личная ссылка на дашборд |
|---|---|---|
| Иса | режиссёр (видит всё) | `…/dashboard/?key=fd37…` |
| Амин | аниматор | `…/dashboard/?key=70f6…` |
| Тимур | аниматор | `…/dashboard/?key=7d55…` |

**Это пропуска сотрудников.** Скопируйте табличку целиком — она нужна мне.
Каждому человеку потом даётся только его строка: ссылки личные, в общий чат
их кидать нельзя.

Если вместо таблички выскочило красное сообщение — пришлите его текстом.

> **Storage трогать не нужно.** Раздел с картинками в Supabase мы не используем,
> бакет создавать не надо — кадры лежат в репозитории проекта.

### Шаг 5. Взять три значения для меня

Самый простой путь — кнопка **Connect** вверху страницы проекта: она сразу
показывает и адрес, и ключ. Полный путь, если нужен: **Settings → API Keys**.

1. **Project URL** — адрес вида `https://xxxxxxxx.supabase.co`.

## 5. План подключения дашборда

> Раздел описательный. Код будет писать отдельный агент; здесь — контракт и
> список того, что придётся поменять.

### 5.1 Библиотека или обычный fetch

**Рекомендую обычный `fetch`, без библиотеки:**

- в `dashboard/index.html` уже есть рабочая функция `saveSupabase()` на `fetch`;
- дашборду нужны четыре запроса, ради этого тянуть библиотеку незачем;
- никакого риска, что сторонний CDN однажды не ответит.

Если позже понадобится Realtime — библиотека доступна: стабильная версия
**supabase-js 2.110.7**, подключается как
`<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"></script>`,
глобальная переменная называется `supabase` (значит, свой клиент так называть
нельзя). Для ESM брать **esm.sh**, а не `+esm` у jsdelivr — у последнего
известный баг с подмодулями.

### 5.2 config.js

Только публичные значения — файл лежит в публичном репозитории:

```js
window.BACKEND_CONFIG = {
  supabaseUrl: "https://xxxxx.supabase.co",
  supabaseKey: "sb_publishable_...",   // публичный ключ, НЕ secret
  seriesId: 1
};
```

Бакет и пути к Storage не нужны: картинки лежат рядом, в самом репозитории.

### 5.3 Четыре запроса

Общие заголовки: `apikey: <ключ>` и `Authorization: Bearer <ключ>`.
CORS настраивать не нужно — Supabase отдаёт `Access-Control-Allow-Origin: *`
принудительно, это даже не конфигурируется.

**1. Кто пришёл** — один раз при загрузке:

```
POST /rest/v1/rpc/whoami      { "p_key": "<из ?key= в адресе>" }
```
Пустой ответ — гость, режим просмотра. Иначе `{name, role, series_ids}`.
Токен обратно не возвращается (проверено).

**2. Список кадров** — версии подтягиваются вложенно одним запросом:

```
GET /rest/v1/frames
    ?select=id,code,scene_n,seq,shot,action,dialogue,refs,prompt_en,status,
            feedback,assignee,is_anchor,final_ok,version_count,updated_at,
            versions(v,provider,full_url,preview_url,prompt_en,generated_at)
    &series_id=eq.1
    &order=scene_n.asc,seq.asc
```

**3. Одобрить / отправить на переделку:**

```
POST /rest/v1/rpc/review_frame
     заголовок: x-review-key: <токен>
     { "p_frame_id": 42, "p_status": "redo", "p_feedback": "Свет слишком холодный" }
```

Токен можно слать и заголовком, и полем `p_key` — работают оба (проверено).
Заголовок предпочтительнее: секрет не попадает в тело.

Ошибки приходят готовым русским текстом, их можно показывать в `showToast()`
как есть: «Нет доступа: ссылка недействительна или отозвана», «Для переделки
нужен комментарий: что именно поправить», «Кадр ещё не сгенерирован — одобрять
нечего».

**4. Дешёвая проверка «что-то изменилось»:**

```
GET /rest/v1/progress?select=total,approved,review,redo,queued,waiting_anchor,updated_at
    &series_id=eq.1
```

### 5.4 Опрос: как не спалить трафик базы

Сейчас в коде `setInterval(autoRefresh, 5000)` тянет весь `frames.json`. Так
оставлять **нельзя**: 166 кадров с промтами каждые 5 секунд — это порядка
100 ГБ в месяц против 5 ГБ бесплатного лимита Supabase. Лимит сгорит за пару
дней, после чего база уйдёт в read-only и дашборд встанет.

Правильно:
- раз в 10–15 с дёргать `progress` (сотня байт);
- полный список перечитывать, только если счётчики или `updated_at` изменились;
- сохранить существующее правило «не обновлять, пока пользователь печатает
  правку» — оно уже есть и работает.

Обратите внимание: этот лимит про **базу**, а не про картинки. Картинки идут с
GitHub Pages, где лимит в 20 раз больше и мягкий.

### 5.5 Картинки — почти ничего не меняется

`full_url` и `preview_url` хранятся как пути относительно корня репозитория
(`images/s01_03_v1.webp`, `previews/s01_03_v1.webp`) — ровно в том же виде, в
каком сейчас лежит `file` в `frames.json`.

Существующая функция `fileUrl()` уже собирает адрес как `'../' + file` и
одинаково работает и на Pages, и локально через `serve.py`. **Её логику можно
оставить как есть** — поменяется только то, какое поле в неё передаётся:
`preview_url` для сетки и `full_url` для клика и кнопки «Скачать».

Это заметно проще, чем было бы со Storage, где пришлось бы собирать абсолютные
адреса с доменом проекта.

### 5.6 Что придётся поменять в существующем коде

| Место | Что сделать |
|---|---|
| `saveSupabase()` | вместо `PATCH /rest/v1/frames?id=eq.X` — вызов `rpc/review_frame`; прямой PATCH база больше не примет |
| `backendMode()` | добавить чтение `?key=` и вызов `whoami`; нет ключа — режим просмотра |
| `findCard(id)` | `id` теперь число, номер кадра «1.3» лежит в `code` — показывать надо `code` |
| `fileUrl()` | логика та же, но в сетку передавать `preview_url`, а по клику `full_url` |
| `STATUS` | добавить `waiting_anchor` → «Ждёт мастер-кадр», отдельный цвет |
| карточка | значок «мастер-кадр сцены» при `is_anchor`, отметка «утверждено режиссёром» при `final_ok` |
| `autoRefresh()` | опрашивать `progress`, полный список — только при изменении |
| селектор сотрудника | сейчас переключатель Амин/Тимур/Иса чисто визуальный; роль должна приходить из `whoami` |

### 5.7 CSP на GitHub Pages — проверено, помех нет

Запросил заголовки живого дашборда напрямую:

```
$ curl -sSI https://genvid25.github.io/amintako/dashboard/
HTTP/2 200
content-type: text/html; charset=utf-8
access-control-allow-origin: *
cache-control: max-age=600
```

**Заголовка `Content-Security-Policy` нет вообще** — ни запросы к
`*.supabase.co`, ни сторонние CDN ничем не ограничены. Утверждения в блогах про
«строгий CSP на GitHub Pages» путают Pages с самим github.com.

Попутно: `cache-control: max-age=600` — страница дашборда кэшируется на 10 минут,
поэтому правки вёрстки появляются у сотрудников с задержкой до 10 минут.

### 5.8 Ловушка для воркера: синхронизация раскадровки

Это про воркер, но обнаружилось при проверке схемы.

Воркер периодически перезаливает кадры из `storyboard.json`. Соблазн написать
«обнови всё, что пришло» — и это молча уничтожит работу людей: кадр, который
аниматор час назад отправил на переделку с комментарием, вернётся в очередь с
пустым `feedback`, и комментарий пропадёт навсегда.

**Так нельзя** (проверено — правка исчезает):

```sql
on conflict (series_id, code) do update
  set status = excluded.status, feedback = '';   -- стирает работу людей
```

**Так правильно** — трогаем только содержание кадра из раскадровки, а решения
людей и работу агента (`status`, `feedback`, `prompt_en`, `assignee`) не
упоминаем вовсе:

```sql
on conflict (series_id, code) do update
  set shot     = excluded.shot,
      action   = excluded.action,
      dialogue = excluded.dialogue,
      sound    = excluded.sound,
      chron    = excluded.chron,
      refs     = excluded.refs;
```

Проверено: описание обновляется, статус `redo` и комментарий остаются на месте.
