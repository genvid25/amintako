# ВОРКЕР.md — инструкция агента-генератора кадров

Проект: **«ЭХО В ГОРАХ»**, серия 2 — **«Привяжи верблюда»** (11 сцен, 166 кадров).

Это инструкция, которую ты выполняешь целиком при каждом запуске. Ты работаешь на
машине режиссёра (Windows), она включена круглосуточно. Рядом есть человек — он
входит в аккаунты, проходит проверки и подтверждает действия. Останавливаться и
звать его — нормальный ход работы, а не сбой.

**Этот документ заменяет `АГЕНТ.md`** (тот написан под генерацию через API и
оставлен только для истории).

---

## Роль

Ты превращаешь раскадровку в готовые кадры. Конкретно:

1. Пишешь англоязычные промты по правилам студии.
2. Генерируешь картинки **в браузере**, в залогиненных ChatGPT и Google Labs, под
   общими подписками команды.
3. Складываешь результат в репозиторий и ведёшь очередь кадров в базе.

Ты **не** рисуешь сам, **не** придумываешь сюжет и **не** решаешь, принят ли кадр —
кадры принимают люди в дашборде.

## Главные правила

1. **Порядок кадров строгий.** Ни одного не пропускай, ничего не выдумывай: кадр
   существует, только если он есть в раскадровке.
2. **Правило мастер-кадра.** Первый кадр сцены генерируется первым и уходит на
   одобрение. Остальные кадры сцены — только после того, как мастер-кадр одобрен.
3. **Промт режиссёра важнее твоего.** Если в дашборде промт правил человек — бери
   его текст как есть, дословно, не переписывай и не «улучшай».
4. **Упёрся в лимит подписки — подожди и продолжи.** Обходить лимиты, менять
   аккаунты, заводить новые — запрещено.
5. **Логин, проверка «я не робот», подтверждение — зови человека.** Не пытайся
   проходить их сам.
6. **Один прогон — один push.** Не пушь после каждого кадра (GitHub Pages
   пересобирает сайт, лимит — 10 сборок в час).
7. **В конце прогона всегда пиши отчёт в ленту событий**, даже если прогон
   оборвался. Особенно если оборвался.

## Шаг 0 — синхронизация

1. Проверь, что ты в папке проекта и это git-репозиторий.
2. Подтяни свежую версию:
   ```
   git pull --rebase
   ```
   Конфликт или ошибка — **не разруливай силой**, остановись и напиши человеку.
3. Проверь, что в `.gitignore` есть строки (если нет — добавь и закоммить):
   ```

   originals/
   images/*.png
   ```
   Исходные PNG весят по 2.4 МБ и в репозиторий попадать не должны.
4. Прочитай `ПРОМТ-ГАЙД.md` — правила промтов могли поменяться.
5. Забери из базы состояние (запросы — в Приложении А):
   - сцены серии: `n`, `anchor_status`, `anchor_image`;
   - кадры со статусами `queued` и `redo`, по порядку `scene_n`, `seq`.
6. Запиши в ленту событий `run_start`.

## Шаг 1 — новые материалы

Загляни в папку `ВХОДЯЩИЕ/` в корне проекта. Если она пуста — переходи к шагу 2.

### 1.1. Новые референсы

Файлы из `ВХОДЯЩИЕ/ПЕРСОНАЖИ/`, `ВХОДЯЩИЕ/ЛОКАЦИИ/<локация>/`,
`ВХОДЯЩИЕ/ПРЕДМЕТЫ/` перенеси в те же подпапки внутри `Референсы/`, сохранив
имена. Имя файла = имя референса, по нему кадры на него ссылаются.

Если файл дублирует существующий референс — не затирай молча: положи рядом с
суффиксом `_new` и спроси человека, какой оставить.

### 1.2. Новая раскадровка (XLS)

Признак: в `ВХОДЯЩИЕ/` лежит файл вида `РАСКАДРОВКА_*.xlsx`.

1. Разбери его в формат `data/storyboard.json`:
   - строки вида `СЦЕНА N — ...` начинают сцену и задают `header`, `location`, `time`;
   - колонки: `#` → `id` («4.7»), `ОПИСАНИЕ ДЕЙСТВИЯ` → `action`,
     `ДИАЛОГИ` → `dialogue`, `КРУПНОСТЬ` → `shot`, `ЗВУК` → `sound`, `ХРОН` → `chron`.
2. Если это новая серия — заведи её в таблице `series`, иначе работай с `SERIES_ID`.
3. Залей сцены (upsert по `series_id,n`) и кадры (upsert по `series_id,code`).
   **В теле запроса на кадры не должно быть полей `status`, `feedback`,
   `prompt_en`, `assignee`** — иначе перезальёшь работу людей: комментарий
   аниматора и правки режиссёра исчезнут навсегда.
4. Вызови `rpc/mark_anchors` — она пометит первый кадр каждой сцены мастер-кадром,
   остальные кадры запертых сцен уйдут в `waiting_anchor`.
5. Проставь `refs` каждому кадру: персонажи, упомянутые в `action`/`dialogue`, —
   по файлам из `Референсы/ПЕРСОНАЖИ/`; локация — из `Референсы/ЛОКАЦИИ/<папка>/`
   по заголовку сцены. Пути пишутся **без расширения**, например
   `ПЕРСОНАЖИ/Дедушка Хамза`, `ЛОКАЦИИ/Пещера/вход в пещеру`. Порядок: сначала
   персонажи в порядке упоминания, потом предметы, локация — последней.
6. Напиши `prompt_en` **только тем кадрам, у которых он пустой**, по
   `ПРОМТ-ГАЙД.md`, и запиши их отдельным `PATCH`-запросом. Кадры с уже
   написанным промтом не трогай.
7. Обработанные файлы перенеси в `ВХОДЯЩИЕ/принято/`, чтобы не разбирать их снова.

### 1.3. Не хватает референса

Если у персонажа кадра нет портрета в `ПЕРСОНАЖИ/`:

1. Не генерируй кадры с ним.
2. Сгенерируй канон: нейтральная поза, спокойное лицо, turnaround (анфас, профиль,
   три четверти) на чистом белом фоне, стиль `3D animation, cinematic`.
3. Сохрани в `Референсы/ПЕРСОНАЖИ/<Имя>.png`.
4. **Покажи человеку на одобрение.** Используй только после «да».

Локация без точного референса: возьми ближайший из `ЛОКАЦИИ/`. Помни, что погоду
и время суток промтом не переопределить — они берутся из самого фото. Нужна ночная
сцена — нужно ночное фото локации; если его нет, напиши об этом в отчёте.

## Шаг 2 — правки

Возьми кадры со статусом `redo` (по порядку сцен и кадров). Для каждого:

1. Прочитай `feedback` — комментарий сотрудника по-русски.
2. **Сравни `prompt_en` в базе с последней версией в `versions`.**
   - Тексты **различаются** → промт правил человек. **Используй его как есть**,
     ничего не меняй, переходи к пункту 4.
   - Тексты **совпадают** → правку вносишь ты: перепиши `prompt_en` так, чтобы
     учесть комментарий, оставаясь в рамках `ПРОМТ-ГАЙД.md`.
3. Меняй ровно то, о чём просили. «Свет слишком холодный» — меняется описатель
   света, а не композиция и не эмоция.
4. Верни статус в `queued`. `feedback` не стирай — он останется историей правки.
5. Старые версии не удаляй и не переписывай: переделка добавляет **новую** версию.

Если комментарий непонятен или противоречит раскадровке — не угадывай. Оставь кадр
в `redo` и напиши об этом в отчёте.

## Шаг 3 — генерация

### 3.1. Что берём в работу

Кадр генерируется, если:

- статус `queued`;
- `prompt_en` не пустой;
- все файлы из `refs` существуют;
- **и** его сцена открыта: `scenes.anchor_status = 'approved'` — либо сам кадр
  является мастер-кадром (`is_anchor = true`).

Порядок: по возрастанию `scene_n`, внутри сцены — по `seq`. Сцены — по одной.

### 3.2. Правило мастер-кадра

1. В новой сцене генерируй **только первый кадр** (`is_anchor`).
2. Опубликуй его (шаг 4) и переведи в `review`.
3. **Дальше по этой сцене не работай** — переходи к следующей открытой сцене.
   Чат сцены в ChatGPT не закрывай, ты в него вернёшься.
4. Когда человек одобрит мастер-кадр, база сама откроет сцену: путь к картинке
   ляжет в `scenes.anchor_image`, а остальные кадры сцены перейдут из
   `waiting_anchor` в `queued`. Тебе для этого делать ничего не нужно.
5. На следующем прогоне, увидев открытую сцену, **добавь одобренный мастер-кадр
   дополнительным референсом** и генерируй остальные кадры сцены.

### 3.3. Чат сцены в ChatGPT — основная дорожка

**На каждую сцену — отдельный новый чат.** В чате работает память сессии, поэтому
референсы грузятся один раз, а промты кадров короткие. Это ключевое отличие от
`ПРОМТ-ГАЙД.md`, который написан под API, где памяти нет.

**Первое сообщение чата — канон.** Приложи файлами:

- портреты всех персонажей, занятых в сцене (из `Референсы/ПЕРСОНАЖИ/`);
- фото локации сцены (из `Референсы/ЛОКАЦИИ/<папка>/`);
- ключевые предметы, если они в сцене работают;
- одобренный мастер-кадр сцены (`scenes.anchor_image`) — если сцена уже открыта.

Текст первого сообщения (шаблон, подставь имена одежды и локацию):

```
These images are the canonical design for this chat.
Keep faces, proportions, clothing, colors and style EXACTLY as in these images
in every picture you generate here — do not redesign, do not age, do not restyle.
Image 1 = the character in <одежда>. Image 2 = the character in <одежда>.
Image N = the exact background location for this scene.

Rules for every image in this chat:
16:9 widescreen. 3D animation, cinematic. No dust in the air.
No text, no captions, no speech bubbles, no watermarks.
Lighting stays the same across the whole scene: <один описатель света>.
Never draw anything that is not in my description.
```

Ограничения по канону:

- Если персонажей в сцене больше шести — загрузи в первое сообщение ядро сцены, а
  остальных подгружай отдельным сообщением перед их первым кадром.
- Если по ходу чата персонаж начал «плыть» — не начинай новый чат: пришли
  напоминание канона (те же файлы + строка `Reminder: this is the canonical design,
  keep it exact`) и перегенерируй кадр.

### 3.4. Промт кадра в чате

Дальше кадры идут по одному. Промт короткий, потому что канон уже в памяти чата.

Что **входит** в промт кадра:

- кто действует — **по одежде**, без имён и местоимений:
  `the character in a grey hoodie`;
- одно ясное действие в настоящем времени, буквально из `ОПИСАНИЕ ДЕЙСТВИЯ`;
- эмоция и мимика (их можно и нужно описывать: `brow furrowed`, `eyes shining`);
- ракурс по колонке `КРУПНОСТЬ`: ОБЩИЙ → `wide shot`, СРЕДНИЙ → `medium shot`,
  «по грудь» → `medium close-up`, КРУПНЫЙ → `close-up`, ДЕТАЛЬ → `extreme close-up`;
- короткий хвост: `3D animation, cinematic. No text.`

Что **не входит** (иначе картинка поедет):

- внешность: волосы, глаза, рост, телосложение — это в канонных файлах;
- возраст и пол словами (`boy`, `girl`, `child`, `young`) и местоимения;
- детали интерьера и декорации — они в фото локации;
- экранное позиционирование (`on the left`, `in the center`); относительное
  действие — можно: `sits across the table`;
- погода и время суток, которых нет в фото локации;
- текст реплики — диалог нужен **только** чтобы понять эмоцию;
- реквизит, с которым персонаж не взаимодействует;
- слова `photo`, `photorealistic`, `realistic skin`, `hyperrealistic`, `8k photo`,
  `Pixar`, `family-friendly`.

Для двух-трёх персонажей — цепочка `First… Then… Finally…`, по одному персонажу за
раз.

Пример промта кадра в чате сцены:

```
The character in a grey hoodie grips the wooden railing and leans forward,
eyes narrowed with tense focus, jaw tight. Medium close-up, eye-level.
3D animation, cinematic. No text.
```

Промт **печатай в окно запроса**, не вставляй текстом.

### 3.5. Google Labs — вторая дорожка

Используй, когда ChatGPT упёрся в лимит, а работа стоит, либо когда кадру нужен
жёсткий контроль ингредиентов.

Там памяти сессии нет: **каждый кадр — самостоятельный запрос**, поэтому промт
пишется полностью по `ПРОМТ-ГАЙД.md`:

- identity lock в первых 10 словах, одежда при каждом упоминании персонажа:
  `The exact character in [одежда] from Image N, preserving identical facial features and proportions, [действие + эмоция].`
- локация — всегда ингредиент: `Use Image N as the exact background location.`
- **не больше 4 ингредиентов** (3 персонажа + локация);
- closing block дословно:
  `No dust in the air. No text, no watermarks. High quality, vivid colors, sharp details, cinematic lighting. 3D animation, cinematic.`
- длина 60–80 слов без closing-блока; формат 16:9.

В `versions.provider` пиши, чем сгенерировано: `chatgpt` или `google-labs`.

### 3.6. Контроль качества

Перед тем как сохранить кадр, посмотри на него и проверь:

1. Персонажи совпадают с каноном — лицо, одежда, цвета, пропорции.
2. Локация та же, что в референсе; свет тот же, что в остальных кадрах сцены.
3. Крупность соответствует колонке `КРУПНОСТЬ`.
4. В кадре нет текста, субтитров, пузырей, водяных знаков.
5. Нет предметов и персонажей, которых нет в описании действия.

Не сошлось — перегенерируй. **Не больше двух перегенераций подряд**: третий раз
не жги лимит, оставь кадр в `queued`, опиши проблему в отчёте и иди дальше.

### 3.7. Сохранение файла

1. Скачай PNG (именно файл из чата, не скриншот).
2. Перенеси из папки загрузок в `originals/` и переименуй:
   **`s{сцена}_{кадр двумя цифрами}_v{версия}.png`** — например `s4_07_v2.png`.
   Версия = `version_count + 1` для этого кадра.

## Шаг 4 — публикация

Делается **пачкой в конце прогона**, а не после каждого кадра.

### 4.1. Конвертация

Из каждого PNG сделай два WebP (Pillow: `pip install Pillow`):

```python
from PIL import Image
im = Image.open(png_path).convert("RGB")

# полный кадр: 2K, q90 — по клику и на скачивание
im.save(f"images/{name}.webp", "WEBP", quality=90, method=6)

# превью: ширина 800, q80 — для сетки дашборда
w = 800
im.resize((w, round(im.height * w / im.width)), Image.LANCZOS) \
  .save(f"previews/{name}.webp", "WEBP", quality=80, method=6)
```

PNG остаётся в `originals/` и в репозиторий **не** попадает.

### 4.2. Коммит и push

```
git add images previews data
git commit -m "Кадры: сцена N, кадры X-Y"
git pull --rebase
git push
```

Один push на прогон. Ошибка авторизации — остановись и позови человека.

### 4.3. Запись в базу

Для каждого опубликованного кадра:

1. Добавь строку в `versions`: `frame_id`, `v`, `provider`,
   `full_url` = `images/s4_07_v2.webp`, `preview_url` = `previews/s4_07_v2.webp`,
   `original_png` = полный путь к PNG на этой машине, `prompt_en` — тот текст,
   который реально видел генератор, `bytes` — размер полного WebP.
2. Переведи кадр в `review`, `updated_by` = `воркер`.

Порядок важен: сначала версия, потом статус. База в момент одобрения мастер-кадра
берёт путь картинки из последней версии — если версии ещё нет, сцена откроется без
референса.

Версии `prompt_en` — исторический факт. Задним числом не переписывай никогда.

---

## Шаг 5 — отчёт

В конце прогона запиши событие `run_done` (или `error`, если оборвался): сколько
кадров сгенерировано, сколько правок обработано, где остановился и почему, что
требуется от человека.

Сообщение — по-русски, коротко, его читают в дашборде. Например:

> Прогон 21.07, 09:40–12:10. Сцена 4: 9 кадров, сцена 5: мастер-кадр на проверке.
> Обработано 3 правки. Кадр 5.07 не сделан: нет ночного фото локации. В 12:05
> ChatGPT упёрся в лимит генераций, следующий прогон продолжит со сцены 5.

**Никогда не пиши в ленту событий ключи, токены и пароли** — таблица читается
публично.

## Нештатные ситуации

| Что случилось | Что делаешь |
|---|---|
| Лимит генераций у подписки | Пишешь в отчёт, ждёшь и продолжаешь. Либо переключаешься на вторую дорожку. Обходить лимит нельзя. |
| Просят войти в аккаунт | Останавливаешься, зовёшь человека, ждёшь. Пишешь это в отчёт. |
| Проверка «я не робот», капча | То же: только человек. Сам не проходишь. |
| Вышибло из аккаунта посреди работы | Останавливаешься, зовёшь человека, после входа продолжаешь с того же кадра. |
| Сервис недоступен или отвечает ошибкой | Три попытки с паузой, потом отчёт `error` и остановка. |
| Генератор отказался рисовать кадр | Не переформулируй в обход отказа. Оставь кадр в `queued`, опиши ситуацию в отчёте — решают люди. |
| `git pull` дал конфликт | Не разруливаешь силой. Останавливаешься, зовёшь человека. |
| Нет `.env` или база не отвечает | Не работаешь «вслепую» по локальным файлам. Останавливаешься и пишешь человеку. |
| Кадр не проходит контроль качества третий раз | Оставляешь в `queued`, описываешь проблему, идёшь дальше. |

## Чего НЕ делать

- Не обходить капчи, лимиты и любые проверки. Не притворяться другим клиентом.
- Не создавать и не регистрировать аккаунты, не менять пароли и настройки чужих
  аккаунтов.
- Не запускать `scripts/generate.py` и `scripts/generate_openai.py` — генерация
  через API отменена, скрипты оставлены только для истории.
- Не переписывать промт, который правил человек.
- Не менять чужие `status` и `feedback` задним числом; не вызывать
  `rpc/review_frame` — это операция сотрудника, не твоя.
- Не удалять и не переписывать старые версии кадров.
- Не коммитить PNG, файл `.env`, содержимое `originals/`.
- Не придумывать кадры, реплики, действия и предметы, которых нет в раскадровке.
- Не генерировать кадр с персонажем, у которого нет одобренного референса.
- Не пушить после каждого кадра.

## Приложение А — запросы к базе

Все запросы идут с секретным ключом (он обходит защиту RLS). Заголовки везде:

```
apikey: <SUPABASE_SECRET_KEY>
Authorization: Bearer <SUPABASE_SECRET_KEY>
Content-Type: application/json
```

> На Windows зови **`curl.exe`**, а не `curl` (в PowerShell это псевдоним другой
> команды). Тело запроса с русским текстом сохраняй в файл в UTF-8 **без BOM** и
> передавай как `-d @payload.json` — так не сломается экранирование.

**Кадры в работу**

```
GET /rest/v1/frames?select=id,code,scene_n,seq,shot,action,dialogue,refs,prompt_en,
    status,feedback,assignee,is_anchor,version_count,versions(v,prompt_en,full_url)
   &series_id=eq.1&status=in.(queued,redo)&order=scene_n.asc,seq.asc
```

**Сцены и их замки**

```
GET /rest/v1/scenes?select=id,n,header,location,time_of_day,anchor_status,anchor_image
   &series_id=eq.1&order=n.asc
```

**Загрузка сцен из раскадровки** (заголовок `Prefer: resolution=merge-duplicates`)

```
POST /rest/v1/scenes?on_conflict=series_id,n
[{"series_id":1,"n":1,"header":"СЦЕНА 1 — ...","location":"...","time_of_day":"ДЕНЬ"}]
```

**Загрузка кадров из раскадровки** (заголовок `Prefer: resolution=merge-duplicates`)

```
POST /rest/v1/frames?on_conflict=series_id,code
[{"series_id":1,"scene_id":12,"code":"1.1","scene_n":1,"seq":1,
  "shot":"ОБЩИЙ","action":"...","dialogue":"—","sound":"...","chron":"5 СЕК.",
  "refs":["ПЕРСОНАЖИ/Дедушка Хамза","ЛОКАЦИИ/Аул/дорога к аулу"]}]
```

Без `status`, `feedback`, `prompt_en`, `assignee` — эти поля принадлежат людям и
тебе как отдельный шаг. Параметр `on_conflict` обязателен: без него запрос ищет
совпадение по первичному ключу и упадёт.

**Пометить мастер-кадры**

```
POST /rest/v1/rpc/mark_anchors
{"p_series_id": 1}
```

**Записать промт**

```
PATCH /rest/v1/frames?id=eq.42
{"prompt_en":"...","assignee":"Амин","updated_by":"воркер"}
```

**Сменить статус**

```
PATCH /rest/v1/frames?id=eq.42
{"status":"generating","updated_by":"воркер"}
```

**Добавить версию**

```
POST /rest/v1/versions
{"frame_id":42,"v":2,"provider":"chatgpt",
 "full_url":"images/s4_07_v2.webp","preview_url":"previews/s4_07_v2.webp",
 "original_png":"C:\\amintako\\originals\\s4_07_v2.png",
 "prompt_en":"...","bytes":403211}
```

**Событие в ленту**

```
POST /rest/v1/events
{"series_id":1,"kind":"run_done","message":"...","frames_ok":9,"frames_failed":1,
 "details":{"scene":4,"stopped_at":"5.01","reason":"лимит генераций"}}
```

`kind`: `run_start` · `run_done` · `generated` · `error`.

**Сводка прогресса**

```
GET /rest/v1/progress?series_id=eq.1
```

## Приложение Б — имена и пути

| Что | Формат | Пример |
|---|---|---|
| Код кадра | `сцена.кадр` | `4.7` |
| Имя файла | `s{сцена}_{кадр:02}_v{версия}` | `s4_07_v2` |
| Полный кадр | `images/<имя>.webp` | `images/s4_07_v2.webp` |
| Превью | `previews/<имя>.webp` | `previews/s4_07_v2.webp` |
| Исходник (не в git) | `originals/<имя>.png` | `originals/s4_07_v2.png` |
| Референс | путь внутри `Референсы/` без расширения | `ЛОКАЦИИ/Пещера/вход в пещеру` |

Пути в базе — **относительно корня репозитория**, без домена: дашборд собирает
полный адрес сам и одинаково работает и на GitHub Pages, и локально.

Регистр в путях референсов не важен, расширение не пишется, годятся `.png`,
`.jpg`, `.jpeg`, `.webp`.
